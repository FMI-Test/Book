## 1) Canonical sync rules (FA / EN / media)

Use these normalization rules to keep Persian and English versions aligned.

| Variant(s) seen in notes | Canonical form to keep |
| --- | --- |
| `go aak`, `GO AAK`, `GOGOGOGOGOAAK`, `gogogo aak` | **Go AAK** |
| `go sync`, `GOSYNC`, `sync gogogo` | **Sync** |
| `nano bananan`, `nana banana`, `babnana`, `nano banana` | **Nano Banana** |
| `Ali yar`, `Ali-Yar`, `Aliyar` | **Aliyar** in prose / `Ali-Yar-...` in filenames |
| `Mumtahen-ul-Mulk`, `Momtahen-ol-Molk`, `ممتحن‌الملک` | Preserve source spelling by language; note variants in metadata |

### Editorial sync policy

- Keep **the same core chronology** in `Aliyar-Fakhran-FA.md` and `Aliyar-Fakhran-EN.md`.
- Let tone vary, but keep **facts, titles, dates, places, and award names** synchronized.
- If an image is used in one language version, it should be either:
  1. used in the other version too, or
  2. documented here with a reason.
- Normalize asset naming before publication; avoid duplicate spellings where possible.

## 7) Five alternatives to Nano Banana (image generation)

| Tool | Pros | Cons |
| --- | --- | --- |
| **GPT-4o Images** | strong prompt following, fast iteration, great for editorial refinement | less stylized than some art-first tools |
| **Midjourney** | beautiful cinematic outputs, great mood and atmosphere | weaker reproducibility / less structured asset control |
| **FLUX / FLUX Pro** | excellent realism and flexible self-hosting options | setup and tuning can be heavier |
| **Ideogram** | strong typography/text-in-image, good posters/cards | more design-oriented than archive realism |
| **Stable Diffusion XL / ComfyUI** | maximum control, local workflows, custom pipelines | highest learning curve, more manual QA |

### Quick rule

- **Fast editorial iteration:** GPT-4o Images
- **Cinematic moodboard:** Midjourney
- **Controlled local pipeline:** FLUX / SDXL

## 8) LLM / workspace options (Copilot, Cursor, Nimotron, etc.)

| Tool / model family | Best for | Pros | Cons |
| --- | --- | --- | --- |
| **GitHub Copilot** | VS Code-native repo work | best editor integration, inline edits, coding flow | less flexible for broad long-form writing than chat-first tools |
| **Cursor** | codebase-wide refactors and agentic edits | strong repo actions, fast editing loops | can drift if prompts are vague |
| **OpenAI / Codex-style workflows** | planning, transformation, long prompts | strong structured reasoning and documentation work | needs careful verification on facts |
| **Claude** | long-form analysis and prose cleanup | strong writing tone and summarization | not always best for IDE-native repo ops |
| **NVIDIA NIM / Nimotron / local stacks** | privacy-sensitive or self-hosted workflows | control, deployment flexibility, local inference paths | more setup, weaker out-of-box editorial flow for some tasks |

### Recommendation for this repo

- **Primary editor:** GitHub Copilot in VS Code
- **Heavy rewrite / second opinion:** Cursor or Claude-style review
- **Private / local experimentation:** NIM / local models

## 9) Automation principle (AAK systems note)

This repo should avoid manual repetition where a stable process can be automated.

### Canonical pattern

1. **Concurrent async generation** where safe
2. **Eventual consistency check** after all outputs land
3. **Garbage collection / cleanup** of duplicates and temp assets
4. **Sync** across FA / EN / media metadata
5. **Consolidation + integrity/alignment test** before publish
6. **Auto-configure and optimize** only after the content model is stable

This is the same design logic behind scalable services: do not repeat a solved pipeline by hand.
